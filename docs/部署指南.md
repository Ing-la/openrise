# 部署指南

本地 Docker 测试与服务器部署的完整流程。

---

## 一、架构概览

| 容器 | 镜像/构建 | 职责 |
|------|-----------|------|
| **nginx** | nginx:stable-alpine | 反向代理、流量分发、静态资源缓存 |
| **frontend** | 从 ./frontend 构建 | Next.js 应用（含 API Routes、Prisma migrate） |
| **db** | postgres:15-alpine | PostgreSQL 数据库 |

### 目录结构

```
ai-web-community/
├── docker-compose.yml    # 编排配置
├── .env.example          # 环境变量模板（Docker 用）
├── frontend/
│   ├── Dockerfile
│   ├── .env.example      # 本地开发用（可选）
│   └── app/api/          # API 路由
└── nginx/
    └── nginx.conf        # 反向代理配置
```

### 持久化

| 数据 | Volume | 说明 |
|------|--------|------|
| 数据库 | `db_data` | PostgreSQL 数据 |
| 用户上传 | `uploads_data` | 头像、课程封面（`public/uploads/`） |

## 二、环境变量

### 两份配置的用途

| 文件 | 用途 | 何时生效 |
|------|------|----------|
| **根目录 `.env`** | docker-compose 变量源 | `docker-compose up` 时注入到容器 |
| **`frontend/.env`** | Next.js 变量源 | `npm run dev` 本地开发时 |

**Docker 部署**：只需根目录 `.env`，frontend 容器不读取 frontend/.env。  
**本地开发**：需要 frontend/.env。

### 根目录 .env（Docker 用）

| 变量 | 作用 | 如何获取 |
|------|------|----------|
| `POSTGRES_USER` | 数据库用户名 | 自行设定 |
| `POSTGRES_PASSWORD` | 数据库密码 | 自行设定，生产环境用强密码 |
| `POSTGRES_DB` | 数据库名 | 自行设定 |
| `NEXTAUTH_SECRET` | JWT 签名密钥，用于会话加密 | 执行 `openssl rand -base64 32` 生成 |
| `NEXTAUTH_URL` | 应用对外访问地址 | 本地：`http://localhost`；生产：`https://你的域名` |
| `RESEND_API_KEY` | Resend 邮件 API 密钥 | [resend.com](https://resend.com) 注册 → API Keys |
| `RESEND_FROM` | 发件人邮箱 | 验证域名后自定义；开发可用 `onboarding@resend.dev` |

### frontend/.env（本地开发用）

| 变量 | 作用 | 如何获取 |
|------|------|----------|
| `DATABASE_URL` | Prisma 连接串 | `postgresql://用户:密码@localhost:5432/库名` |
| `NEXTAUTH_SECRET` | 同上 | 同上 |
| `NEXTAUTH_URL` | 本地 dev 用 | `http://localhost:3000` |
| `RESEND_*` | 同上 | 同上 |
| `NEXT_PUBLIC_TURNSTILE_SITE_KEY` | Turnstile 站点密钥 | [Cloudflare Turnstile](https://dash.cloudflare.com/?to=/:account/turnstile) |
| `TURNSTILE_SECRET_KEY` | Turnstile 私钥 | 同上 |

## 三、本地 Docker 测试

### 前置条件

- 已安装 [Docker Desktop](https://www.docker.com/products/docker-desktop/)（Windows/Mac）或 Docker Engine + Docker Compose（Linux）
- 80 端口未被占用（或修改 `docker-compose.yml` 中 nginx 的端口映射）

### 1. 准备环境变量

```bash
cd ai-web-community

# 若根目录尚无 .env，复制模板
copy .env.example .env   # Windows
# cp .env.example .env   # Linux/Mac
```

若不存在 `.env`，docker-compose 会使用 yml 中的默认值，可直接运行；建议复制并保留以便管理。

### 2. 启动并构建

```bash
docker-compose up -d --build
```

首次构建约需 3–5 分钟。

### 3. 检查容器状态

```bash
docker-compose ps
```

预期：db、frontend、nginx 均为 running。

### 4. 验证

| 操作 | 预期结果 |
|------|----------|
| 访问 http://localhost | 打开首页 |
| 访问 http://localhost/api/health | 返回 `{"status":"ok"}` |
| 访问 http://localhost/courses | 打开案例库 |

### 5. 功能测试清单（可选）

- [ ] 首页、案例库、课程详情可正常打开
- [ ] 注册、登录、个人中心
- [ ] 创建课程、添加章节、小节（视频/Markdown）
- [ ] 上传头像、课程封面

### 6. 测试完毕

```bash
# 停止并移除容器（保留 volume 数据）
docker-compose down

# 若需同时删除数据库和上传数据
docker-compose down -v
```

## 四、服务器部署

### 部署前检查

| 项目 | 说明 |
|------|------|
| 域名 | 若用域名，需在 DNS 中解析到服务器 IP |
| 防火墙 | 开放 80（HTTP）、443（HTTPS） |
| HTTPS | 当前为 HTTP；如需 HTTPS，需在 nginx 中配置 SSL 证书 |

### 1. 本地：提交并推送到 GitHub

```bash
cd ai-web-community

git add .
git commit -m "部署相关更新"
git push origin main
```

**务必先推送**，服务器再拉取最新代码。

### 2. SSH 登录服务器

```bash
ssh user@your-server-ip
```

### 3. 拉取代码

```bash
# 首次部署
git clone https://github.com/你的用户名/ai-web-community.git
cd ai-web-community

# 后续更新
git pull origin main
```

### 4. 配置环境变量

```bash
cp .env.example .env
nano .env   # 或 vim .env
```

**生产环境至少修改：**

| 变量 | 说明 |
|------|------|
| `POSTGRES_PASSWORD` | 强密码 |
| `NEXTAUTH_SECRET` | `openssl rand -base64 32` 生成 |
| `NEXTAUTH_URL` | `https://你的域名` |
| `RESEND_API_KEY`、`RESEND_FROM` | 如需邮件功能 |

### 5. 构建并启动

```bash
docker-compose up -d --build
```

### 6. 验证

- 访问 `http://服务器IP` 或 `https://你的域名`
- 访问 `/api/health` 检查健康状态

## 五、常用命令

```bash
# 启动
docker-compose up -d --build

# 查看日志
docker-compose logs -f

# 查看某服务日志
docker-compose logs -f frontend

# 停止
docker-compose down

# 重新构建
docker-compose up -d --build --force-recreate
```

## 六、常见问题

### 1. 端口被占用

修改 `docker-compose.yml` 中 nginx 的 `ports`，例如 `"8080:80"`，然后访问 `http://localhost:8080`。

### 2. frontend 构建失败

```bash
docker-compose build frontend --no-cache
```

常见原因：网络问题导致 npm install 失败，可重试。

### 3. frontend 容器反复重启

```bash
docker-compose logs -f frontend
```

检查是否有 `Cannot find module` 等依赖缺失错误。

### 4. 数据库连接失败

```bash
docker-compose ps db
docker-compose logs db
```

确认 db 容器健康后再启动 frontend。

### 5. 注册/登录异常

检查 `NEXTAUTH_URL`：Docker 环境下应为 `http://localhost`（或通过 nginx 的端口）；生产环境为 `https://你的域名`。

### 6. 上传图片报错（JSON 解析失败或“请求异常”）

**可能原因与排查**：

| 情况 | 排查步骤 |
|------|----------|
| Nginx 413（文件过大） | 在 `location /api` 中增加 `client_max_body_size 5m;`，重启 nginx |
| 请求未到达 Next.js | 浏览器 F12 → Network → 找到 `/api/upload` 请求，看 **Status** 与 **Response**。若为 404/502，多为宿主机 Nginx/Apache 拦截，需将 `/api` 反向代理到 Docker |
| 会话或权限 | 确认已登录；若为 500，查看 `docker-compose logs frontend` |
| 宿主机反向代理 | 若服务器 80 端口由宿主机 Nginx 接管，需在其 `location /api` 中 `proxy_pass http://127.0.0.1:容器端口` |

### 7. 镜像里包含环境变量吗？

不包含。变量在容器启动时由 docker-compose 注入。

## 七、后续扩展

- **HTTPS**：在 `nginx/` 下添加 SSL 证书，修改 `nginx.conf` 启用 443 端口
- **Redis**：如需缓存或会话，可在 `docker-compose.yml` 中新增 service
